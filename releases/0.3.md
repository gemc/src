# Release notes

- Added Linux binary `gemc` tarballs to the development deployment workflow.
  The archives contain the installed executable, support files, selected smoke
  tests, `gemc.env`, and a generated Geant4 data installer.
- Added automated binary tarball release and installation tests on base OS
  images using runtime packages only. These tests install Geant4 datasets into
  the unpacked tarball and run `gemc -v` plus smoke tests after the tarballs are
  attached to the `dev` release.
- Cleaned up CI job labels, opted JavaScript-based workflow actions into
  Node.js 24, and adjusted CodeQL to avoid the Actions overlay-base fallback
  warning.
- Added a configurable splash-screen display duration and set the GEMC GUI
  startup splash to remain visible briefly before closing.
- Fixed setup-tab geometry reloads so the Volumes tab is rebuilt from the
  reloaded detector geometry.
- Reopened the Geant4 GUI scene during setup-tab reloads so reloaded geometry is
  displayed before running events.
- Rebuilt setup-tab geometry immediately for display/Volumes updates, then
  reinitialized through Geant4 before Run so MT workers use a fresh world.
- Synchronized setup-tab system selection back into runtime options so pressing
  Run after Reload uses the same geometry shown in the display.
- Deferred Geant4 initialization when starting `gemc -gui` without a selected
  system, preventing synthetic ROOT-world visualization state before setup-tab
  reloads.
- Fixed GUI Geant4 output splitting by normalizing line endings before board
  logging instead of using a newline regular expression.
- Allowed `gemc -gui` to open with an empty setup tab when the default
  `gemc.db` is not present, instead of exiting with SQLite error code 209.
- Allowed the Volumes tab to start empty when no Geant4 world has been built,
  avoiding a fatal pre-window geometry-map lookup.
- Generated the installed examples SQLite database during `meson install` so
  the GUI setup tab is populated after a normal install.
- Initialized the Geant4 run manager during the GUI pre-run geometry handoff so
  `BeamOn` is not ignored after setup-tab reloads.
- Rebuilt setup-tab run geometry from fresh system descriptors so preview
  reloads do not leave factory-populated systems that duplicate ROOT volumes at
  `BeamOn`.
- Isolated ROOT linkage to the ROOT gstreamer plugin so the main `gemc`
  executable can run without ROOT shared libraries.
- Fixed GUI crash (SIGABRT in `G4ToolsSGSceneHandler::GetOrCreateNode`) when
  reloading geometry after running events: `refreshGeometryTree` now tracks
  whether a viewer already exists and skips `/vis/open` on subsequent reloads,
  avoiding a second viewer creation that triggers a synchronous ToolsSG refresh
  before the Qt widget is ready.
- Fixed hits not recorded after GUI setup-tab reload or on the second and
  subsequent BeamOn calls: introduced a `digiplugins_need_reload` flag in
  `GDetectorConstruction` so `loadDigitizationPlugins()` (which clears the
  shared digitization-routines map) runs only when geometry actually changed,
  not on every `ConstructSDandField()` invocation during routine BeamOn
  re-initialization.  In Geant4 task-parallel MT, `RunInitialization()` can
  overlap with workers already in their event loop; the previous code cleared
  the shared map while workers were between `BeginOfRunAction` (which saw the
  map populated and opened streamers) and `EndOfEventAction` (which then found
  the map empty), causing a fatal "no digitization routine registered" error.
  The flag is reset to `true` by `reload_geometry()` and
  `prepare_geometry_for_run()` so that the next `ConstructSDandField()` on the
  master thread refreshes the map exactly once per geometry change.
- Fixed crash ("no digitization routine registered for collection X") when
  running events after switching geometries in the GUI setup tab: Geant4's
  `G4SDManager` accumulates sensitive detectors across geometry reloads and
  fires `Initialize()` on all of them at the start of every event, even those
  no longer attached to any volume in the current geometry.  These stale
  detectors create empty hit collections that appear in `G4HCofThisEvent`.
  `EndOfEventAction` now skips any collection whose SD name is not in the
  current digitization-routines map with a non-fatal warning instead of
  terminating the process, so current-geometry hits are still processed
  correctly.
- Cleared the digitization-routines map at the start of `loadDigitizationPlugins()` so that
  `emplace()` cannot silently reuse stale plugin objects from a prior load.
- Cleared accumulated GUI visualization event drawings on setup-tab geometry
  reloads so hits and trajectories from a previous run are not redrawn with the
  newly loaded system.
- Made the setup-tab experiment/system column resize after initial population,
  matching the readable sizing already applied after user selections.
- Made the GUI follow the active Qt palette more consistently, including
  theme-aware setup tables, log/search widgets, geometry-tree color controls,
  navigation SVG buttons, and splash-screen rendering.
- Added theme-specific rasterized splash images for the built-in GEMC
  architecture splash and a configurable scale plus palette-aware border.
- Removed the need for GEMC-specific environment variables in Python example
  workflows: the API runs through the `pygemc` module, and users only need the
  executable in `PATH`.
- Updated the Cherenkov optical example with neutral demonstration radiator names:
  `lowIndexRadiator`, `mediumIndexRadiator`, and `highIndexRadiator`.
- Simplified the Test workflow so it runs directly inside the matching
  `ghcr.io/gemc/g4install:<geant4>-<os>-<tag>` job container instead of
  generating and building a temporary Dockerfile.
- Consolidated Test, Sanitize, and Deploy matrix generation in
  `ci/distros_tags.sh`; the shared matrix source now emits normal build,
  sanitizer, and manifest matrices with consistent image, Geant4, runner, and
  container fields.
- Reduced sanitizer-build static linkage by allowing GEMC internal libraries to
  follow Meson's `default_library` option and selecting shared GEMC libraries
  for sanitizer builds, avoiding static-TLS pressure when tests load plugins
  with `dlopen`.
- Added y-vs-x analyzer plotting support in `pygemc` and used it for the
  Cherenkov homepage plots.
- Reorganized all source modules under a `gemc/` subdirectory and adopted
  `<gemc/module/header.h>` namespaced `#include` directives throughout; symlinks
  in `src/gemc/` make the same form work at build time as well as after install.
- Set white as the default Geant4 viewer background color.
- Added a `description` field to `GVolume` for documenting detector components;
  the description is shown in the geometry-tree properties panel when a volume
  is selected.
- Added "Centre" and "Twinkle" action buttons to the geometry-tree left navigation
  bar: "Centre" re-centres the Geant4 viewer on the selected volume; "Twinkle"
  briefly highlights it by toggling visibility.
- Added color-picker buttons for the annotation scale and frame in the display
  utilities panel, replacing fixed colors with user-configurable choices.
- Replaced all four display-option toggle buttons (Anti-aliasing, Hidden lines,
  Auxiliary edges, Field lines) with theme-aware SVG icons; a single SVG file
  covers both checked and unchecked states â€” `refresh_svg_icons()` substitutes
  palette-correct background and foreground colors at render time on every
  toggle or theme change, removing the need for per-state image files.
- Added an "Inspect `<volume>`" button in the geometry-tree right panel: clicking
  it opens the selected logical volume in a dedicated viewer window using the same
  driver as the active viewer, with a white background and a centred 2D
  volume-name label; focus is automatically restored to the original viewer
  window after the new scene is flushed.
- Replaced the separate `punit`, `aunit`, and `vunit` fields in `gparticle` with
  direct `value*unit` notation on each kinematic field (e.g. `p: 4*GeV`,
  `theta: 23*deg`, `vz: -10*cm`). A plain number without a unit is still accepted
  and falls back to MeV for momentum, `deg` for angles, and `cm` for vertex
  coordinates, with a warning logged. This aligns the particle-gun syntax with the
  rest of GEMC's unit conventions and fixes a crash when `value*unit` expressions
  were used directly in `gparticle` YAML blocks.

## Compatibility notes

- **`gparticle` unit fields removed**: the `punit`, `aunit`, and `vunit` fields are
  no longer recognized. Replace them by attaching the unit directly to each value:

  ```yaml
  # before
  gparticle:
    - name: e-
      p: 2300
      punit: MeV
      theta: 23.0
      aunit: deg
      vz: -10.0
      vunit: cm

  # after
  gparticle:
    - name: e-
      p: 2300*MeV
      theta: 23*deg
      vz: -10*cm
  ```

  A plain number without a unit is still accepted and defaults to MeV / deg / cm
  (with a logged warning), so existing steering cards that omit the unit fields
  continue to work.

- Added `gemc::collectPluginOptions(int argc, char* argv[])` declared in `gemc_options.h` and
  implemented in `gemc_options.cc`. Before the main `GOptions` parsing constructor is called in
  `main()`, GEMC scans every YAML file listed in `argv` for `gsystem` entries, resolves each
  `<name>.gplugin` using the plugin search path, and probes for an optional
  `extern "C" GOptions* definePluginOptions()` symbol. Options declared there are merged into
  the schema so plugin-specific options appear in `gemc <config>.yaml -h`, are saved to the
  configuration snapshot, and can be set from the command line or YAML alongside core options.

  macOS, reference counting achieves the same effect. Plugin verbosity keys are registered by
  passing the channel name to the single-argument `GOptions` constructor inside
  `definePluginOptions`, adding `verbosity.<plugin>` to the schema independently of the global
  `gdigitization` verbosity.
